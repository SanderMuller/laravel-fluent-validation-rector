# 0.19.1 — `RuleSet::from()` descent + docblock-narrow multi-pass diagnosis

## Overview

Patch release driven by hihaho 0.19.0 dogfood. Two findings:

1. **P0 — silent bail on `RuleSet::from([...])` return shape.** Real-world `rules()` methods following the `laravel-fluent-validation` recommended idiom return `RuleSet::from([...])` rather than a bare array. The rector currently visits, sees the `MethodCall` wrapper around the array literal, walks away without folding **and without logging**. Same class as the 0.13.3-era silent-path issues.

2. **P1 — `UpdateRulesReturnTypeDocblockRector` skips post-fold `'*' => FluentRule::array()->children([Array_])` shape on real multi-rector pipelines, despite the predicate accepting that shape on static input.** Hihaho saw `[fluent-validation:skip] ... value at key '*' is not a FluentRule chain (shape: MethodCall)` after 0.19's fold ran. A static-input fixture (`narrows_after_groupwildcard_fold_children.php.inc`) confirms the predicate handles `children([Array_])` correctly when the input is hand-written. Root cause is downstream of the predicate — multi-pass timing, name-resolution scope, or a fold-emit AST shape that diverges from the hand-written equivalent. Spec calls this **diagnose-before-patch**.

## 1. Scope

### In scope

- **P0**: descend into `RuleSet::from($args[0])` argument when it's an `Array_` literal, and run the existing `groupRulesArray()` over it. Two emit modes: silent re-fold (no skip log) on success, descriptive skip-log entry on failure.
- **P1**: build an end-to-end fixture proving the skip on the real multi-rector pipeline. If reproduced, find the actual root cause (`getName()` resolution scope, fold-emit StaticCall shape, traversal order). Patch only after diagnosis.
- **§4.1 — Cross-fold-path interaction matrix.** Promote the cross-case invariant (added retroactively for 0.19's `mixed_literal_and_const_wildcard_keys` codex find) to a permanent §4 subsection: every new fold path must ship at least one fixture per non-trivial intersection with each existing fold path.

### Out of scope (reopen on consumer signal)

- **`RuleSet::make()->merge([...])` chain descent.** No consumer using this idiom yet (hihaho doesn't, collectiq doesn't). A `merge()` chain descent is a much bigger ask — instance-method walking, multiple potential fold targets in a single chain, `merge(self::baseRules())` indirection. Reopen if a real codebase surfaces this shape with `'*.' . CONST` siblings inside.
- **`RuleSet::compileToArrays([...])` inline-at-validate-site shape.** Per collectiq dogfood: 7 hits across 5 Filament Livewire pages, all at `validate()` call sites rather than `rules()` method bodies. Out of scope for the current rector entry-point design (gates on `rules(): array | rules(): RuleSet` method bodies). Would require a "rector-finds-rules-array-anywhere" expansion, much larger than 0.19.1.
- **`RuleSet::create()` and other static factories** — verified against vendor: `RuleSet::create()` doesn't exist. `make()` is zero-arg. `compile*()` family is internal pipeline. `from()` is the only consumer-facing static factory taking an array.

## 2. P0 — `RuleSet::from()` descent

### 2.1 Current behavior (broken)

The rector's traversal in `refactor()` calls `traverseNodesWithCallable($method, fn($inner) => …)` with a closure that bails on the very first line:

```php
if (! $inner instanceof Return_ || ! $inner->expr instanceof Array_) {
    return null;
}
```

For `return RuleSet::from([...]);`, `$inner->expr` is a `MethodCall` (or `StaticCall` for the FQN `\SanderMuller\FluentValidation\RuleSet::from()` form), not an `Array_`. The closure returns null, the rector visits the next node, and the wrapped array literal is never seen.

### 2.2 New behavior

Treat `RuleSet::from(<Array_>)` as an alias for the wrapped `Array_` for fold purposes. The descent helper resolves either:

```
return [...]                          // existing path: $inner->expr is Array_
return RuleSet::from([...])           // new path: $inner->expr is StaticCall(RuleSet, 'from'), arg 0 is Array_
return \SanderMuller\FluentValidation\RuleSet::from([...])   // FQN variant
```

…to the inner `Array_` and folds it. The fold-applied wrapper stays in place — the StaticCall's args[0] is mutated in-place to the new array shape.

### 2.3 Bail conditions

Each emits a descriptive skip-log entry under `=actionable`:

- `RuleSet::from()` argument is not an `Array_` (e.g. `RuleSet::from($injected)`, `RuleSet::from(self::baseRules())`).
- More than one argument to `RuleSet::from()` (defensive — current signature is single-arg, but future API drift shouldn't silent-bail).
- Method body has multiple `Return_` nodes or non-trivial control flow before the `RuleSet::from()` return — fold target ambiguous.

### 2.4 Why descend, not bail-with-log

Two-fix options were proposed:
1. Descend and fold (this spec).
2. Bail with explicit log: "wrapped in `RuleSet::from()` — return bare array if you want fold."

(2) was rejected because `RuleSet::from()` is the canonical recommended return shape per the `laravel-fluent-validation` docs. Bail-with-log makes the consumer choose between idiomatic `rules()` and rector benefit. Descending fold preserves both.

## 3. P1 — Diagnose docblock-narrow multi-pass skip

### 3.1 Reproduction plan

Build an end-to-end fixture under `tests/Parity/Fixture/UpdateRulesReturnTypeDocblockRector/` (or a new integration-test directory) where:

- Input: bare-array `'*.' . CONST` siblings (the 0.19 main-feature shape).
- Both `GroupWildcardRulesToEachRector` AND `UpdateRulesReturnTypeDocblockRector` configured in the rector pipeline.
- Expected output: the array folded to `'*' => FluentRule::array()->children([CONST => …, …])` AND the `@return` annotation narrowed to `array<string, FluentRuleContract>`.

If the test fails the same way hihaho's synthetic did (skip log on the docblock-narrow rector after the fold), the bug is reproduced and we can debug.

### 3.2 Candidate root causes (to be narrowed by reproduction)

1. **`getName()` resolution scope.** The fold emits `new StaticCall(new Name('FluentRule'), …)` — short name. The predicate calls `$this->getName($current->class)` and compares against `FluentRule::class` (FQN). Rector's `getName()` may need a `use` statement in scope to resolve the short name to the FQN. Hand-written fixtures get a name resolver pass; fold-emitted AST may not, depending on traversal order.
2. **Multi-rector pipeline ordering.** If `UpdateRulesReturnTypeDocblockRector` runs in the same pass as `GroupWildcardRulesToEachRector`, it may visit the pre-fold AST. The skip message naming key `'*'` argues against this — pre-fold there's no `'*'` key — but worth ruling out.
3. **Fold-emit AST shape diverges from hand-written.** A hand-written `FluentRule::array()->children([…])` has `Name(['FluentRule'])` for the StaticCall class. The fold-emitted version may use a different node class, attribute set, or parent reference.

### 3.3 Decision: patch only after diagnosis

Do not widen the predicate. The static-input fixture (already added) confirms the predicate is correct on the documented shape. Adding more cases pre-diagnosis would either be a no-op or paper over the real bug.

If diagnosis turns up #1 (name resolution): fix is a narrowing of the predicate's class match — accept either FQN or short-name `FluentRule`, since the fold emits short-name nodes that haven't been through the name resolver. Single-line fix.

If #2 (pipeline ordering): fix is in the rector pipeline configuration or the docblock-narrow rector's gating, not the predicate.

If #3 (fold-emit AST shape): fix is in the fold's emit code, or the predicate's traversal-tolerance.
