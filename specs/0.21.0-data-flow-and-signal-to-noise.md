# 0.21.0 — unsafe-parent data-flow tightening + skip-log signal-to-noise pass

## Overview

Minor release. Two interlocking heuristic-surface fixes plus an
orthogonal skip-log convention sweep. Driven by 0.20.x dogfood:
- mijntp 0.20.1 confirmed "concerning case" on `ServiceProviderAdminPage`
  — closure-scope fix alone didn't address the underlying issue;
  cross-directory ancestor-chain over-attribution is the real bug.
- mijntp 0.20.1 audit findings #2-#4 (line numbers, name-the-key,
  expression breadcrumbs) deferred to this release alongside the
  data-flow work since it reshapes some skip messages anyway.
- 0.20.0 + 0.20.1 messages already deferred tier-3 detail-map
  population ("subtle test regressions through static-state
  interactions; tackling alongside data-flow tightening").

This release is **1.0 RC critical-path**: until the unsafe-parent
heuristic is data-flow-aware, the hedged-generic skip is the
de-facto majority case for Filament/Livewire-heavy codebases —
forming a "diagnostic too vague to act on" first impression for
cold consumers.

## 1. Scope

### In scope

- **#38 Data-flow tightening**: tighten `collectUnsafeParentClass`
  so it only flags a parent unsafe when an array op's input
  variable provably traces back to a `parent::*()` return value.
  Replaces the current "any parent-call + any array-op coexist in
  the same method" heuristic.
- **#37 Tier-3 detail-map AST population**: once data-flow
  tightening drops false positives, populate the unsafe-parent
  detail map from the project-wide filesystem scan path so the
  named-descendant skip-log branch fires reliably (not just when
  the descendant happens to be visited in the same process). Was
  attempted in 0.20.1 but caused unrelated test regressions; with
  data-flow tightening the surface stabilizes enough to retry.
- **#44 Skip-log signal-to-noise pass**:
  - Append `:LN` (file:line) to skip-log entries via
    `Node::getLine()` so editor click-to-open works.
  - "Name the key" convention sweep across all rectors that visit
    array entries — currently inconsistent (`UpdateRulesReturnTypeDocblockRector`
    names keys, `SimplifyRuleWrappersRector` doesn't).
  - Expression-text breadcrumbs: `(shape: $this->passwordRules())`
    instead of `(shape: MethodCall)`; `index 4 (`spread expr text`)`
    instead of bare `index 4`.

### Out of scope (reopen on consumer signal)

- **`#[FluentRules]` audit-assertion bypass for unsafe-parent**
  (deferred from 0.20.2). Conflicts with pinned design in
  `skip_attributed_method_when_subclass_manipulates.php.inc` —
  needs a separate design-review cycle. Not bundled here because
  the data-flow tightening removes the need for the bypass on
  the dominant false-positive surface anyway.
- **Per-attribute Livewire overlap-skip emit** (`#31`). Different
  rector, different concern (cardinality vs heuristic accuracy).
  Bundled into 1.0 RC checklist as a separate item.

## 2. Resolved questions

All three resolved 2026-04-28 via three-lens peer audit (hihaho deep-
dogfood, mijntp skip-log signal-to-noise, collectiq cold-consumer +
drained-surface). Lens-diversity invariant satisfied (three independent
consumer lenses converged on each OQ).

### OQ #1 — data-flow trace depth → (a) Direct only

Empirical evidence across three consumer codebases:
- **hihaho**: 12 `parent::rules()` usages, all spread (`[...parent::rules(), …]`)
  or method-chain (`parent::rules()->modify(…)`). Zero `array_merge(parent::rules(), …)`.
  Zero aliased `$base = parent::rules(); …` shapes (verified via
  `grep -rE '\$\w+\s*=\s*parent::rules'`).
- **mijntp**: zero `parent::rules()` references at all (the
  `ServiceProviderAdminPage` skip was pure heuristic over-reach).
- **collectiq**: drained surface; no `parent::rules()` shapes.

(a) "direct only" covers 100% of observed real-world surface at lowest
implementation cost. Reopen-on-consumer-signal pattern reserves
(b)/(c) for the future if a consumer reports an aliased false negative
post-1.0.

#### Non-targets of the data-flow trace (hihaho clarifications)

Two AST shapes that LOOK like array manipulation on `parent::rules()`
but are NOT classified as array ops by `matchArrayManipulationOp` —
pinning here so the design intent is explicit:

1. **Spread inside `Array_`** — the 9 hihaho Player-request shapes:
   ```php
   return [...parent::rules(), 'extra' => …];
   ```
   `Spread` AST node isn't `FuncCall` / `StaticCall` / `Unset_` /
   array-dim `Assign`. Spread is argument-unpacking semantics, not
   array mutation; the parent's return value is consumed verbatim
   into the new array literal. No mutation; no false positive.

2. **Method chain on `parent::*()`** — the 3 hihaho Request-extension
   shapes:
   ```php
   return parent::rules()->modify(…)->merge(…);
   ```
   `parent::rules()->modify()` is a `MethodCall` on a `StaticCall`.
   `matchArrayManipulationOp` only matches specific function-call
   shapes (`array_*`, `in_array`, `collect`, `Arr::*`, `unset()`,
   array-dim assignment). Generic MethodCalls aren't classified as
   array ops. Combined with 0.20.1's closure-scope skip stopping
   traversal into the closure arg of `->modify(…)`, the chain itself
   contributes zero false-positive surface.

Both are already-correct behavior; explicit non-targets pinned here
so the next contributor doesn't accidentally widen the heuristic to
catch them.

### OQ #2 — tier-3 detail-map state-leak isolation → (c) Scope-narrow to `src/`

Test fixtures under `tests/` are intentional shape demonstrations of
unsafe patterns; they shouldn't contribute to the production heuristic
state. (c) matches semantic intent: production heuristic informed by
production code only. Two reasons (cold-consumer lens):

1. **Perf**: walking `tests/` on every rector run is pure overhead for
   files that by design shouldn't shape production heuristics.
2. **Correctness/precision**: leaking test-fixture FQCNs into the
   project-wide map risks a similarly-named `src/` class getting
   mis-skipped because its name collided with a deliberate-unsafe
   fixture. False positives possible, hard to debug.

Edge case: consumers shipping test-helper packages with production
code under `tests/`. <5% of consumer surface; addressable via
existing rector path config. Default-(c) serves the 95%.

### OQ #3 — line-number emit format → `(/path:42)`

Format adopts `Node::getStartLine()` of the offending node, appended
inside the existing parens-wrapped path for byte-stability with
existing grep workflows. Verified across:

- JetBrains terminal: clickable, strips trailing `)`. ✓
- VS Code terminal: clickable, resolves correctly. ✓
- vim quickfix: `:[0-9]+\)` regex extracts cleanly. ✓
- iTerm2 cmd-click: works. ✓

Granularity:
- **Per-key/per-item skips**: offending `ArrayItem->getStartLine()`.
- **Class-wide skips**: `Class_->getStartLine()`. Multi-rector skips
  on the same class produce stable line numbers so `grep -F ":42)"`
  workflows find related entries cleanly.
- **Method-body skips** (e.g. SimplifyRuleWrappers): chain's first
  node `getStartLine()`.

Consistent default: **first relevant node's `getStartLine()`**.

Skipped: `:42:` colon-bracketed (GCC-style) — the trailing `:` reads
ugly in prose context, and column numbers aren't meaningful for
rector skip diagnostics where the offending node spans an expression,
not a token.

## 5. Cross-fold-path interaction matrix (per specs/PROCESS.md §1)

#38 is a heuristic tightening (replaces an existing detection), not
a new fold path. No cross-fold-path matrix required — but a
"before vs after" matrix on the heuristic decision boundary is
valuable:

| Shape                                                                   | 0.20.x | 0.21.0 |
|-------------------------------------------------------------------------|--------|--------|
| `array_merge(parent::rules(), [...])` direct                            | flag   | flag   |
| `$base = parent::rules(); array_merge($base, [...])`                    | flag   | OQ #1  |
| `parent::canAccess()` + unrelated `array_map($users)` (ServiceProvider) | flag   | drop   |
| `parent::rules()` + unrelated `array_map($otherVar)`                    | flag   | drop   |
| Closure-arg `return RuleSet::from(...)->modify(closure)` (UpdateQuestion) | drop   | drop   |
| Ancestor with no rules-bearing surface (ServiceProviderAdminPage's 9 descendants) | drop (0.20.2) | drop |

The matrix shows 0.21.0's heuristic is strictly tighter than
0.20.x's — same true positives, fewer false positives. No
behavior regression risk for true unsafe shapes.
