# 1.0 RC — SemVer commitment cut

## Overview

The 1.0 release commits the public API surface to SemVer 2.0. Every
symbol in `PUBLIC_API.md` becomes frozen until 2.0; every behavior
commitment (skip-log paths, line format, env var vocabulary, wire keys)
becomes a contract whose break is a MAJOR-bump-only event.

The 0.21.x cycle satisfied saturation criteria (PROCESS.md §7) for
the heuristic surface across the three-lens panel (hihaho deep-
dogfood, mijntp skip-log signal-to-noise, collectiq cold-consumer +
drained-surface). N=2 stopping rule met: mijntp "1.0 RC ready from my
side" on 0.21.0; hihaho + collectiq converged on 0.21.1. The
remaining 1.0 RC work is **scope-locking**, not behavior-shaping —
forced dispositions on deferred items + removal of
deprecation shims + final docs tightening.

This spec exists primarily to **force dispositions** on every
backlog item floated as "1.0 RC". Each gets one of: in-scope (ship in
RC), defer-to-1.x (file as backlog with reopen-on-signal trigger), or
out-of-scope (won't do). No item stays ambiguous.

## 1. Scope

### In scope

- **Drop deprecation shims at root namespace.**
  `src/Diagnostics.php` and `src/RunSummary.php` are `class_alias`
  shims pointing at `Internal\Diagnostics` / `Internal\RunSummary`.
  Both PHPDoc-tagged "Removal slated for 1.0" since 0.20.0. Delete
  the shim files; the canonical classes stay under `Internal\`.
- **Mark `LEGACY_FQN_STANDARD_RULES_ANNOTATION_BODY` deprecated**
  (per Resolved §2 — OQ #1 = (b)). Add `@deprecated since 0.22.0 —
  removal slated for 2.0` PHPDoc to the constant + the recognition-
  loop branch. Keep the behavior intact through 1.x. Removal is a
  separate 2.0 cycle, NOT 1.0 RC scope.
- **#45 Skip-log version stamp.** Header currently emits the
  package version via a runtime resolution path. Mijntp finding #5:
  the version stamp drifts when consumer bumps the package without
  clearing the cached log. Source the version from a single
  `Package::VERSION` constant (or composer-derived equivalent); pin
  the resolution path in tests.
- **#47 PUBLIC_API.md "Inspecting fixtures" polish.** Collectiq
  round-4 findings: anchored fixture path, named test runner class,
  trim the `cd` step. Three small edits to the existing section.
- **PUBLIC_API.md behavior-commitments audit.** Read the doc end-to-
  end against the actual 0.21.x behavior. Every implicit
  commitment (skip-log path, log filename, env var values, wire-
  key strings, line-format slot order) gets either explicit listing
  or explicit "implementation detail" exclusion. Cold-consumer lens
  (collectiq) verifies post-edit.
- **README final polish.** Same end-to-end read against 0.21.x.
  Update Known Limitations table to reflect drained shapes (the
  Filament/Livewire false-positive class is gone). Removal of any
  pre-RC "subject to change" caveats — 1.0 is the commitment.

### Defer to 1.x (reopen-on-signal)

- **#37 Tier-3 detail-map AST population.** 0.20.1 attempt caused
  static-state regressions; 0.21.0 data-flow tightening drained
  most of the motivating false-positive surface (mijntp's
  ServiceProviderAdminPage shape, Filament/Livewire). Post-0.21.0
  saturation panel reports zero open false negatives on the
  named-descendant skip-log branch. Reopens if a consumer reports
  cross-directory ancestor-chain over-attribution post-1.0.
- **#31 Per-attribute Livewire overlap-skip emit.** Refactor to
  emit one skip per overlapping `#[Validate]` attribute instead of
  one skip per class. Cardinality fix; not heuristic accuracy.
  No consumer has reported the current single-emit shape as
  unactionable. Reopens if a consumer with N>5 overlapping
  attributes per class reports the single-emit shape obscures
  which attribute caused the overlap.
- **Magic-constant config deprecation cycle.** Wire-key and
  constant-name surfaces are dual-listed in PUBLIC_API.md (constants
  AND literal-string wire keys). The deprecation cycle would
  collapse to one canonical form. Belongs in 1.x (PHPDoc
  `@deprecated`) → 2.0 (removal). Not 1.0 RC scope; documenting
  here so it doesn't get rebundled.
- **`LEGACY_FQN_STANDARD_RULES_ANNOTATION_BODY` removal.** Per
  Resolved §2 — OQ #1 = (b), the constant + recognition branch get
  a `@deprecated since 0.22.0` PHPDoc tag in 1.0 RC scope but the
  removal itself is a 2.0 cycle. Reopens (i.e. the removal lands)
  alongside the magic-constant deprecation cycle in 2.0.

## 2. Resolved questions

### OQ #1 — LEGACY annotation-body deprecation cycle → (b) Deprecate in 1.x, remove in 2.0

Resolved 2026-04-28. The `LEGACY_FQN_STANDARD_RULES_ANNOTATION_BODY`
constant is `protected` on a trait (consumers don't import) but the
*behavior* (passive idempotency on FQN-form annotations from
pre-0.20.x rector runs) is observable. Some consumers may have
committed FQN-form annotations; cutting the recognition in 1.0
would silently rewrite their docblocks on first 1.0 run.

**Resolution**: add `@deprecated since 0.22.0 — removal slated for
2.0` to the constant + recognition branch. Keep the behavior
through 1.x. Bundle the removal into the same 2.0 cycle as the
magic-constant deprecation.

**Why (b) over (a)**: the rewrite cost on first run is low *per
consumer* but the change is silent — no diagnostic emits, no
warning. SemVer's job is to eliminate exactly this class of silent
behavior shifts. Deprecation cycle costs nothing (PHPDoc-only) and
gives consumers an explicit signal.

### OQ #3 — RC cadence → (b) Ship 0.22.0 RC, cut 1.0 after saturation re-converge

Resolved 2026-04-28. PROCESS.md §7 N=2 saturation criteria are
satisfied for the 0.21.x heuristic surface, but the RC scope-locking
work (shim removal, LEGACY annotation-body deprecation, PUBLIC_API.md
tightening) is itself a new emit-surface change. PROCESS.md §7.3
(subsurface clock resets on new emit) requires one more dogfood pass
before counting toward 1.0 stopping rule.

**Resolution**: ship 0.22.0 with the RC scope-lock work; run the
three-lens panel on 0.22.0; cut 1.0 after that pass converges. 1.0
itself is a tag-only release — no behavior change beyond 0.22.0.

**Why (b) over (a)**: cutting 1.0 directly from 0.21.x would mean the
shim removal + deprecation tag land in the 1.0 release itself, which
is the SemVer commitment cut. Better to land them in 0.22.0 (still
pre-SemVer, where breaking changes are permitted by policy) and have
1.0 be tag-only. Risk-asymmetric: if 0.22.0 surfaces a finding, the
fix lands in 0.22.1 without affecting 1.0 timing; if 1.0 itself
surfaces a finding, the fix needs a 1.0.1 patch on a tag that's
already SemVer-committed.

### OQ #2 — Heuristic-boundary commitments → (c) Hybrid one-paragraph carve-out

Resolved 2026-04-28. PUBLIC_API.md gets a short
"Heuristic boundaries (implementation detail)" section: one
paragraph + a tight bullet list of categories (detection
heuristics, classification confidence / trace depth, skip-text
reason content, fixture coverage, diagnostic granularity), no
case-by-case enumeration of every shipped heuristic.

**Why (c) over (a)**: (a) commits to a category-by-category
enumeration that has to be kept in sync with every new heuristic
the rector ships. (c) describes the *shape* of what's
implementation detail (heuristic decision rules, confidence,
trace depth, etc.) without listing each one — durable across
future heuristic additions without doc maintenance churn.

**Why (c) over (b)**: pure-implicit (b) leaves the conflation
between "skip-log line format committed" and "the heuristic that
decides what gets emitted is committed" intact. Cold consumers
have hit this exact reading; one paragraph eliminates the
ambiguity at low doc cost.

The drafted markdown in §3 (now superseded by §4 Phase 3 task list)
serves as the implementation reference. Final wording lands during
Phase 3 (PUBLIC_API.md tightening).

### OQ #4 — Trait-FQN cross-package commitment → (a) Keep current implicit commitment

Resolved 2026-04-28. PUBLIC_API.md "Trait FQNs (referenced by
trait-add rectors)" section stays as-is: implicit commitment that
upstream renames cascade with this rector's MAJOR cycles.

**Why (a) over (b)**: (b) overpromises across a package boundary —
a security/correctness rename upstream would force either a
rector MAJOR bump for behavior the rector didn't change, or an
explicit-commitment violation. (a) keeps the coordination
implicit (same-author, same-cadence), which is how it actually
works.

**Why (a) over (c)**: per-package constant indirection
(`TraitNames::HAS_FLUENT_RULES`) doesn't help — the rector emits
the literal FQN into consumer source code; the constant only
exists at rector-runtime, not in the consumer's post-conversion
codebase.

**Why (a) over (d)**: (d)'s "not coordinated" phrasing risks
cold-consumer misread as "rector may emit broken code". The
implicit "renames upstream cascade" wording in current
PUBLIC_API.md is honest enough without the negative-framing
risk.

No PUBLIC_API.md change for trait-FQN section in 0.22.0.

## 5. Findings

### Phase 1 (2026-04-28)

- **No `tests/InternalAuditTest::PUBLIC_CLASSES` update needed.** The
  shim files (`src/Diagnostics.php`, `src/RunSummary.php`) carried
  `@internal` PHPDoc tags so they always passed the audit's
  "non-public symbol must have @internal" branch — they were never
  listed in `PUBLIC_CLASSES`. Post-deletion, the iterator simply
  doesn't discover them; no test edit required.
- **`config/config.php` referenced the root-namespace shim.** The
  rector/extension-installer entry point called
  `SanderMuller\FluentValidationRector\RunSummary::registerShutdownHandler()`
  — a path that resolved through the deprecation shim. Updated to the
  canonical `Internal\RunSummary` FQN as part of this phase.
  Implication: 0.20.0 introduced the `Internal\` move but kept the
  config entry point on the shim path; 0.22.0 closes that gap.

### Phase 2 (2026-04-28)

- **Version-resolution path: keep `InstalledVersions`, drop the
  `Package::VERSION` constant alternative.** The current path
  (`Composer\InstalledVersions::getPrettyVersion()`) is already
  centralized in `Diagnostics::packageVersion()` and is deterministic
  per-process. A hard-coded `Package::VERSION` constant would require
  bump-time discipline (forget = stale) without solving any drift
  scenario beyond what InstalledVersions covers — both are subject to
  identical opcache-staleness windows since both load from
  vendor-tree PHP files at autoloader time. Local-dev resolution is
  cleaner with InstalledVersions (returns `dev-main` instead of an
  outdated tagged constant).
- **Stale public docblock.** `Diagnostics::skipLogHeader()`'s
  docblock claimed the version was "read from the package's own
  composer.json via `__DIR__/../composer.json`" — actual code uses
  InstalledVersions. Fixed in this phase.
- **Mijntp finding #5 reframed.** Original "drift on bump" framing
  presupposed a missing-truncation scenario; investigation shows the
  session-stale truncation logic does fire correctly per-process. The
  actionable deliverable here is the regression-pin test
  (`SkipLogHeaderTest`) — it locks the resolution path so a future
  refactor (composer.json read, constant, env var) breaks the test
  rather than silently shipping.

### Phase 3 (2026-04-28)

- **Verbose-mode env var commitment was over-specific.** PUBLIC_API.md
  pre-edit listed five accepted values: `off`, `actionable`, `1`,
  `true`, `all`. Code reality: any non-empty non-`actionable` value
  resolves to `all` (legacy permissive fall-through). Tightened doc to
  three canonical values (`off`, `actionable`, `all`) plus `1` /
  `true` as legacy synonyms — preserves the actual contract without
  committing the permissive fall-through as API surface.
- **No README "Known limitations" entry to remove.** Spec called for
  removing the Filament/Livewire false-positive class (drained by
  0.21.0), but that limitation never landed in README — it was an
  internal heuristic that got fixed before any limitation entry
  shipped. No-op for this task.
- **Namespace-structure doc drift.** PUBLIC_API.md said "deprecation
  shims at the root will be removed in 1.0"; updated to reflect
  removal in 0.22.0 (this release).
- **`PSR-4 cwd` phrase was inaccurate.** Removed from skip-log file
  paths section — PSR-4 is autoloading, not path resolution.
- **Heuristic-boundaries section landed per OQ #2 = (c).** One short
  paragraph + tight category bullet list (5 categories: detection
  boundaries, classification confidence, skip-text reason content,
  fixture coverage, diagnostic granularity). No case-by-case
  enumeration of shipped heuristics.
- **#47 fixtures polish landed.** Replaced `git clone` + `cd` block
  with direct GitHub tree links (anchored URLs), named the
  `ParityTest` runner class explicitly, removed redundant `cd` step.
