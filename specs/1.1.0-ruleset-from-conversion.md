# 1.1.0 — RuleSet::from descent for converter rectors

## Overview

`ValidationStringToFluentRuleRector` and `ValidationArrayToFluentRuleRector`
currently anchor on `Return_` whose `->expr` is `Array_`. A method whose
return is `RuleSet::from([...])` (the canonical sister-package wrapper)
does NOT match the anchor — string and array-form rules inside the
wrapper silently skip with no diagnostic.

`GroupWildcardRulesToEachRector` already descends into `RuleSet::from(...)`
correctly via `isRuleSetFromCall()` + `extractArrayFromRuleSetFrom()`
helpers (added 0.19.1). This release extends the same descent pattern
to the two converter rectors so pre-conversion shapes (string rules,
array-form rules) get folded inside the wrapper the same way
post-conversion FluentRule chains already do.

The failure-mode shape is identical to the 0.22.x cross-rector
silent-partial-config class: consumer asks for "convert my string
rules" by configuring the converter; rules inside `RuleSet::from(...)`
silently don't convert; no error fires. This is real Q1-side
correctness coverage under hihaho's refined silently-breaks-vs-
silently-misses-new-features discriminator — the consumer's request
("convert these rules") doesn't land for a documented sister-package
shape.

Surfaced as a post-1.0 finding (2026-04-29) via direct user inspection
of mijntp's `AttachUserToCompanyAction` codebase. The action exhibits
the canonical broader-context shape: `$safeInput = RuleSet::from([...])
->check($data)->safe()` in an action method body, NOT inside a
`rules(): RuleSet` FormRequest method.

## 1. Scope

### In scope

- **Extend `ValidationStringToFluentRuleRector` to descend into
  `RuleSet::from([Array_])`.** Currently anchors on `Return_->expr =
  Array_`; new anchor adds `Return_->expr = StaticCall(RuleSet, from,
  [Array_])` AND `Expression_->expr = (anything containing
  StaticCall(RuleSet, from, [Array_]) as a sub-expression with the
  Array_ accessible)`. The sub-expression case is broader-context: an
  inline assignment + method chain like
  `$safe = RuleSet::from([...])->check($data)->safe()`.
- **Same descent extension for `ValidationArrayToFluentRuleRector`.**
- **Shared helper extraction.** Move `isRuleSetFromCall()` +
  `extractArrayFromRuleSetFrom()` from `GroupWildcardRulesToEachRector`
  into a new `Concerns/DescendsIntoRuleSetFromWrapper.php` trait
  consumed by all three rectors. Single source of truth for the
  descent semantics.
- **Skip-log on non-literal RuleSet::from arg.** Mirror GroupWildcard's
  existing skip emit (`'RuleSet::from() argument is not a literal
  array — fold target must be a statically-determinable Array_;
  consumer audit required'`). Reused verbatim for the converters so
  consumers see one consistent skip-reason wording across all three
  rectors.

### Out of scope (defer to follow-up cycle if signaled)

- **`InlineResolvableParentRulesRector` descent into RuleSet::from.**
  The inline-resolvable rector's anchor is similar in shape (visits
  `parent::rules()` merge patterns). RuleSet::from descent there is a
  natural follow-up but separate scope — file as 1.x backlog if
  consumer signal warrants.
- **`SimplifyFluentRuleRector` / `SimplifyRuleWrappersRector` descent
  into RuleSet::from.** These rectors operate on already-converted
  FluentRule chains. RuleSet::from already protects the FluentRule
  chains inside it from the converters; the post-conversion
  simplification rectors operate at FluentRule-chain level and don't
  need RuleSet::from awareness.
- **`Validator::make($data, RuleSet::from([...])->toArray())` pattern.**
  If `->toArray()` is appended to materialize an array, the inner
  array shape is lost at the call site (returns array, not RuleSet).
  Out of scope — the consumer is past the wrapper at that point.

## 2. Resolved questions

### OQ #1 — Anchor scope → (b) Broad

Resolved 2026-04-29 (user confirmation). The converter rectors descend
into `RuleSet::from([Array_])` static calls **wherever they appear in
PHP source** — action methods, service classes, controller actions,
job handle methods, FormRequest `rules(): RuleSet` returns, anywhere.
The static-call signature `StaticCall(RuleSet, from, [Array_])` is
structurally specific enough to self-gate against false positives;
the surrounding-class-qualification gate the current converters use
(FormRequest descendant / fluent-trait / Livewire / `#[FluentRules]`)
is bypassed for this descent path.

**Why (b) over (a)**:
1. Sister-package docs canonicalize the broader form
   (`RuleSet::from([...])->check($data)->safe()` is the documented
   inline-validation pattern outside FormRequest contexts).
2. Mijntp's `AttachUserToCompanyAction` originating finding IS the
   broader-context shape — action method, not FormRequest. Real-world
   consumer surface.
3. The descent target's static-call signature self-gates; no class
   qualification needed.

### OQ #2 — Bail-string positioning → (a) Source-position-preservation

Resolved 2026-04-29 by existing convention. The current array
converter already preserves bail position per `array_mixed_ordering.php.inc`:

```
input:    ['bail', 'required', new CustomRule(), Rule::requiredIf(...), 'string', 'max:255']
output:   FluentRule::string()->bail()->required()->rule(new CustomRule())->rule(Rule::requiredIf(...))->max(255)
```

Bail and required preserve source position relative to each other
(both before chain-method tails like `max()`). The descent into
`RuleSet::from([...])` reuses the existing array-conversion logic
unchanged; bail position-preservation is an emergent property of
that reuse, not a new behavior choice. No spec departure from
established convention.

### OQ #3 — Method-chain on RuleSet::from preservation → (a) Inner-only descent

Resolved 2026-04-29 by spec recommendation. The
`->check($data)->safe()` chain (and any other method chain on
`RuleSet::from([...])`) is sister-package API contract — collapsing
or rewriting the chain would change runtime behavior (loses the
validation step or shifts return shape). Strict inner-Array_
replacement keeps semantics unchanged; consumer-visible diff is
only the inner array literal converting from string-rule / array-form
shape to FluentRule-chain shape. Pinned by fixture §4.5.

## 5. Findings

### Codex adversarial review post-implementation (2026-04-29)

Codex review on the green-CI 1.1.0 commit surfaced a high-severity
correctness finding before tag cut: `isRuleSetFromCall` originally
matched the bare short-name `'RuleSet'` as a fallback alongside the
package FQCN. Inherited verbatim from `GroupWildcardRulesToEachRector`'s
narrow `rules()` return path (where misuse was bounded), the bare-name
fallback became a real false-positive surface once wired into the
broad-scope `StaticCall` visitor — any global-namespace `RuleSet`
class would have been treated as the sister-package wrapper and had
its inner array rewritten.

**Fix applied pre-tag**: `isRuleSetFromCall` now matches against the
fully-qualified name only (`SanderMuller\FluentValidation\RuleSet`).
PhpParser's NameResolver attaches `namespacedName` attributes that
resolve plain imports, aliased imports (`use ... as VR`), and inline
FQCNs all to the same FQCN string — so the FQCN-only check covers
the legitimate consumer surface without exposing the
bare-name-shadow false-positive risk. Documented in the trait's
PHPDoc with explicit examples of what matches vs what doesn't.

**New regression-pin fixture**: §4.7 (added post-Codex). Aliased
import (`use SanderMuller\FluentValidation\RuleSet as ValidationRules;
ValidationRules::from([...])`) — the descent fires correctly, fold
applies. Pre-fix this would have continued working under the bare-
name fallback only by accident (NameResolver returns the FQCN for
aliases). Post-fix it works by the FQCN check directly. The fixture
documents the intended-behavior contract on aliased imports
explicitly, so future refactor that breaks alias resolution surfaces
as a test failure.

The originally-drafted §4.7 "non-package shadow" fixture was dropped:
namespaced shadow classes (e.g. `App\Domain\RuleSet`) don't exercise
the bare-name fallback path because NameResolver returns the FQCN.
Only global-namespace `RuleSet` references would trip the fallback,
and that surface is hard to construct cleanly inside Rector's
test-fixture format. The PHPDoc on `isRuleSetFromCall` documents
the edge case explicitly as part of the matcher's contract.

### Pre-implementation (2026-04-29)

- **Surfaced via post-1.0 user inspection of mijntp consumer.** The
  finding shape is the same Q1-side silent-partial-config class as
  the 0.22.x cross-rector cycle, but applied to a different surface
  (RuleSet::from descent in converters, not cross-rector
  configuration sharing). Reinforces the broader pattern: "wherever
  the rector's anchor surface is narrower than the documented
  consumer pattern, silent-partial-application surfaces post-tag."
- **Cross-fold-path matrix per PROCESS.md §1**: see §6 below.
- **Lens-scope refinement applied retroactively**: pre-1.0 audits
  saturated under "consumer-shape stability" lens scope (hihaho's
  byte-identical chain) which by construction did not exercise
  RuleSet::from descent — hihaho's actual rector.php was already on
  bare `return [...]` shapes. Mijntp's broader-codebase scan caught
  it first under their own lens but didn't surface it as RC-blocking
  because the impact was bounded. Post-1.0 surfacing IS the lens-
  scope refinement's prediction operationalized: "saturation on
  narrowly-scoped specialty doesn't generalize to the broader surface
  the lens claims to cover." Banking as additional second-cycle
  confirmation evidence for the candidate-pen entry.

## 7. Versioning rationale

Per [README versioning policy](../README.md#versioning-policy):

> **MINOR (x.Y.z)** — additive, non-breaking:
> - Wider match conditions on an existing rector (transformations now
>   apply to shapes that previously fell through)

This release widens the converter anchors to descend into
`RuleSet::from(...)` wrappers — shapes that previously fell through.
Strictly MINOR. Consumer-visible diff for codebases using the
RuleSet::from pattern: rules inside the wrapper now convert to
FluentRule chains where they previously stayed as string/array shapes.

Bump from 1.0.0 → 1.1.0 on release.
