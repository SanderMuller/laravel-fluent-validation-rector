# 1.2.0 — Livewire `#[Rule]` + `HasFluentValidation` compose-conflict warning

## Overview

`ConvertLivewireRuleAttributeRector` gains a layer-2 warning emit
that fires when a Livewire class with property-level
`#[Rule]` / `#[Validate]` attributes has an ANCESTOR using
`HasFluentValidation` (or `HasFluentValidationForFilament`). The
two rule-source mechanisms compose silently:
`HasFluentValidation::getRules()` returns only the fluent `rules()`
method's output and does NOT consult the Livewire attribute
pathway. `validateOnly($prop)` for an attribute-bound property
silently returns `[]` — no exception, no warning, no log. Worse,
rector-side conversion would generate a child `rules()` method
that overrides the parent's, silently dropping parent-owned
fields.

**Direct-trait use on the class itself is intentionally NOT flagged**
(narrowing applied during 1.2.0 candidate review; see Codex finding
in §7). The direct-trait shape converts cleanly: the rector either
merges the attribute rule into a local `rules()` array, or installs
one — no clobber, no silent-ignore post-convert. Bailing on direct-
trait use would block a working transformation.

Surfaced post-1.1.0 from a real consumer 2FA-flow production bug
where `validateOnly('phoneNumber')` returned `[]` for a property
declared as `#[Rule(['required', 'phone:NL'])]` on a class whose
abstract parent used `HasFluentValidation`. Phone validation was
silently bypassed; caught by manual code review, not by the rector.

This is the cross-package equivalent of the intra-package
`warnDenylistedAttributedMethods` pattern: layer-2 detection of a
silent compose-conflict that the consumer's reasonable mental
model wouldn't anticipate. Same shape, different surface.

## Why 1.2 ship direct (not candidate-pen)

Three reasons:

1. **Candidate-pen gates evidence-of-shape, not severity-of-impact.**
   The N=2 stopping rule applies to shapes whose reality is
   uncertain. Here the shape is confirmed real (production bug).
   Withholding the fix waiting for a second consumer would misuse
   the candidate-pen mechanism.
2. **Fix mirrors `warnDenylistedAttributedMethods` verbatim.** Code-
   side correctness extension to a sibling cross-package case,
   not a novel methodology requiring multi-cycle validation.
3. **Q1-side correctness** under hihaho's refined silently-breaks-
   vs-silently-misses-new-features discriminator. The consumer
   wrote `#[Rule([...])]` and expected validation; got empty array.
   Configured behavior stops doing what consumer asked for. Q1
   ships.

## 1. Scope

### In scope

- Layer-2 warning emit in `ConvertLivewireRuleAttributeRector`
  detecting Livewire `#[Rule]` / `#[Validate]` property attribute +
  ANCESTOR class using `HasFluentValidation` (or
  `HasFluentValidationForFilament`) trait. Direct-trait use on the
  class itself is intentionally not flagged (see overview).
- Per-property emit cardinality (one skip-log entry per offending
  property; same shape as existing per-attribute analysis).
- Verbatim skip-log message naming the offending property + both
  compose-resolution options (move rule into trait's `rules()`, or
  remove the trait if Livewire-attribute validation is intended).
- Detection runs unconditionally (BEFORE the existing
  parent-final-rules safety check) so the warning fires even when
  conversion would otherwise bail or succeed.
- Static dedup keyed on class FQCN — same class doesn't emit
  duplicate warnings as the rector visits multiple times.
- Fixture pinning the verbatim wording + skip-log test data set
  entry.

### Out of scope

- Auto-rewrite that moves attribute-bound rules into the
  `HasFluentValidation` trait's `rules()` method. The right fix
  depends on the consumer's intent (Livewire attribute path vs
  fluent-trait path); rector can't pick. Diagnostic only.
- Detection of the inverse compose-conflict (`HasFluentValidation`
  trait + `$this->validate([...])` explicit call). The validate-call
  shape is already partially handled by the
  `KEY_OVERLAP_BEHAVIOR` configuration; deferred to 1.x backlog if
  consumer signal warrants.

## 3. Implementation

Single phase. New private method `warnLivewireFluentTraitComposeConflict`
in `ConvertLivewireRuleAttributeRector`, mirroring
`warnDenylistedAttributedMethods` shape:

1. Static-dedup map keyed on class FQCN (mirror existing pattern).
2. Walk class properties; check each for Livewire `#[Rule]` /
   `#[Validate]` attribute via existing `DetectsLivewireRuleAttributes`
   helpers.
3. For each property carrying the attribute: check
   `aliasAwareAncestorUsesTrait($class, HasFluentValidation::class)` and
   `aliasAwareAncestorUsesTrait($class, HasFluentValidationForFilament::class)`.
   Ancestor-only — direct-trait use on `$class` does not match.
4. If ancestor-trait detected: emit `logSkip` with verbatim message
   naming the property.

Call site: `shouldProcessClass` entry, BEFORE the existing
`hasAnyLivewireRuleAttribute` check. Mirror pattern: warning fires
unconditionally; existing flow continues.

Skip-message text (verbatim):

> property `$<name>` carries `#[Livewire\Attributes\Rule]` (or
> `#[Validate]`) but an ancestor class uses `HasFluentValidation`
> trait — the attribute is silently ignored at runtime
> (`validateOnly()` returns `[]` for this field) and rector-side
> conversion would override the parent's `rules()` method,
> dropping parent-owned fields. Either move the rule into the
> parent's `rules()` method, or remove the trait from the parent
> if Livewire-attribute validation is intended.

## 7. Findings

### Pre-implementation (2026-04-29)

- Originating bug: mijntp 2FA setup flow.
  `SetupSmsTwoFactorAuthentication::send()` called
  `validateOnly('phoneNumber')` and got `[]`; `phoneNumber`
  validation was silently bypassed. Caught by manual code review,
  not by rector. Security-adjacent (2FA flow → phone-number
  capture without server-side rule application).
- The existing `parent class declares final rules() method;
  cannot override — skipping to avoid fatal-on-load` skip is
  correct (prevents fatal-on-load); the absence of the layer-2
  compose-conflict warning is what allowed the silent-attribute-
  ignore class to ship.
- mijntp's fix is to inline the rule into the parent's
  `HasFluentValidation`-trait `rules()` method. Layer-2 warning
  would have surfaced the bug before merge.

### Codex review on the 1.2.0 candidate (2026-04-29)

Pre-tag adversarial review flagged the initial detection scope
as overbroad: the original
`classUsesFluentValidationTrait` helper used
`currentOrAncestorUsesTrait`, which matched **direct-trait use on
the class itself** in addition to the inherited shape. But the
direct-trait shape converts cleanly:

- Class with own `rules()` method: rector merges the attribute
  rule into the existing array. Trait's `getRules()` reads the
  merged result — no silent-ignore.
- Class without own `rules()` method: rector installs one
  containing the attribute rule. Same outcome.

Bailing on direct-trait use blocked a working transformation
(turning a previously fixable class into a hard skip). The
narrowing replaces `currentOrAncestorUsesTrait` with
`aliasAwareAncestorUsesTrait`, scoping the bail to the
inherited-rules-clobber risk only. The positive fixture in §4.2
pins the convertible direct-trait path so a future regression
re-broadening the detection produces a test failure.

### Cross-consumer probe candidate (post-1.2.0)

Collectiq has Livewire-attributed components. Single-cycle
post-1.2.0 dogfood probe to grep their codebase for the same
shape — either zero collectiq instances (clean shape isolation,
mijntp-only-so-far) or surface a second-consumer-hit (broader
shape than mijntp-specific). Worth a dogfood-cycle ask if
collectiq has cycle.
