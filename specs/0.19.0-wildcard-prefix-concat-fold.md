# 0.19.0 — `'*.' . CONST` wildcard-prefix concat fold

## Overview

Widen `GroupWildcardRulesToEachRector` to fold sibling rule keys
shaped `'*.' . CONST` into a new output form
`'*' => FluentRule::array()->children([CONST => …, CONST_2 => …])`
with the `ClassConstFetch` preserved verbatim as the children-array
key (NOT string-literalized).

Driven by 0.17.0/0.17.1 production dogfood: 4 hihaho
`JsonImportValidator` subclasses (`JsonInteractionGroup`,
`JsonInteraction`, `JsonQuestion`, `JsonSubtitle`) use this shape
exclusively for their wildcard rules. Pre-0.19 they hit the existing
`concat key too complex to parse for grouping` skip-log entry; post-
0.17.1 those entries are subsumed under the
`shape-changing rector skipped on Validator subclass` gate (because
they're attribute-only Validator subclasses), but the underlying
parser limitation persists for non-Validator-subclass consumers
that use the same shape.

This is a parser-AND-emitter widening, not just `parseConcatKey()`.
The existing `<class-const> . '<dotted-suffix>'` shape and the new
`'<dotted-prefix>' . <class-const>` shape have STRUCTURALLY
DIFFERENT fold semantics.

---

## 1. Current state

### Existing supported shape: `<class-const> . '<dotted>'`

```php
'rules_with_const_prefix' => [
    self::INTERACTIONS . '.*.type' => FluentRule::string()->required(),
    self::INTERACTIONS . '.*.name' => FluentRule::string()->required(),
],
```

Fold output:

```php
'interactions' => FluentRule::array()->each([
    'type' => FluentRule::string()->required(),
    'name' => FluentRule::string()->required(),
]),
```

The const value (`'interactions'`) becomes the wildcard's parent
key. The suffix-dotted-portion becomes the inner sub-key inside
`each([...])`.

### New shape: `'*.' . <class-const>` (peer-shared, 2026-04-27)

```php
'rules_with_literal_prefix' => [
    '*.' . InteractionGroup::NAME => FluentRule::string()->required(),
    '*.' . InteractionGroup::COLOR => FluentRule::string()->nullable(),
],
```

Target fold output:

```php
'*' => FluentRule::array()->children([
    InteractionGroup::NAME => FluentRule::string()->required(),
    InteractionGroup::COLOR => FluentRule::string()->nullable(),
]),
```

The literal `'*.'` is shared across all sibling keys → fold target
is a top-level `'*' => …` entry. The const becomes a NESTED key
inside `children([...])`, **preserved as a `ClassConstFetch` AST
node, not string-literalized**. Stringifying the const value
would lose source readability and break refactor safety: a
consumer renaming `InteractionGroup::NAME` from `'name'` to
`'name_v2'` should see the rector-produced output update too,
which only happens if the AST node references the const.

### Why this is a new fold output

The existing `each([...])` form takes `<sub-key-string> => <rule>`
pairs. The new form takes `<ClassConstFetch> => <rule>` pairs, and
uses `children()` not `each()`:

- `each(<rule>)` — apply this rule to every wildcard match
- `each(<array>)` — apply each entry of the array to its
  corresponding sub-key under the wildcard
- `children(<array>)` — describe nested children of a wildcard
  match where the children's sub-keys are constant-driven

`children()` exists on `ArrayRule` (verified in vendor). It
accepts a string-keyed array; ClassConstFetch keys evaluate at
runtime to those strings.

### Out of scope for 0.19

Multi-wildcard three-operand concats
(`<const> . '.*.literal.*.' . <const>`) and trailing-wildcard
rule keys (`'... .*'`). These appear in real-world consumer code
(e.g. hihaho's `SaveInteractionRequest`) and currently bail
silently in `GroupWildcardRulesToEachRector`. Reopen as 0.20
candidate with its own fixture share if dogfood signal warrants.
The 0.19 P2 widening is single-wildcard, two-operand only.

Tangential audit follow-up (separate issue, not blocking 0.19):
the silent-bail paths in `GroupWildcardRulesToEachRector` should
be verified to log skip entries for multi-wildcard cases the
parser rejects — analogous to the 0.13.3 skip-log hardening on
the trait-insertion path. Sample case: `SaveInteractionRequest`
has multi-wildcard rule keys but doesn't appear in the skip log.
File a 0.18.x or 0.19.x patch entry once reproduced.

## 2. Parser changes

### Two-shape recognition in `parseConcatKey()`

Today the parser walks `Concat` parts left-to-right and bails when
it sees a `String_` BEFORE any `ClassConstFetch`. The shape it
already accepts is `[ClassConstFetch, String_]`. Widen to also
accept `[String_, ClassConstFetch]` — the literal-prefix-then-const
form — and return a result whose `kind` field discriminates
between the two output paths:

```php
return [
    'kind' => 'const_prefix_string_suffix',  // existing
    'prefixExpr' => $constFetch,
    'suffix' => '.dotted.suffix',
    // …
];

// vs

return [
    'kind' => 'string_prefix_const_suffix',  // new
    'prefix' => '*.',
    'constExpr' => $classConstFetch,
];
```

The `kind` field drives the indexer + grouper + emitter
branches. No behavior change for existing `kind` consumers.

### Three-operand handling: explicitly NOT in scope

Hihaho also has `$this->prefix . '*.' . CONST` shapes — these
appear ONLY in constraint arguments (`->excludeIf($field, …)`,
`->prohibitedUnless($field, …)`), NEVER as rule keys. The
GroupWildcard rector touches rule keys only, not constraint args.
Three-operand parsing stays out of scope. The parser bails on
Concat chains with > 2 parts of the new shape; the existing
2-part `kind: 'const_prefix_string_suffix'` continues to handle
its multi-string-suffix flatten (which is the only multi-part
case it accepts today).

### Edge case bail conditions

Per peer's fixture-matrix suggestions:

**(a) All-same-const-host** — primary case. Fold normally.

**(b) Mixed-const-host** — different FQCNs across sibling keys
(`'*.' . InteractionGroup::NAME` + `'*.' . ExternalInteraction::TYPE`
in the same `rules_with_literal_prefix` array). Accept; the
`children([...])` body holds heterogeneous ClassConstFetch keys,
which is fine — Laravel resolves them to strings at runtime
regardless of declaring class. Document this as supported in the
spec; no behavior delta vs case (a) at the AST level.

Sub-case: same const NAME across different hosts with different
VALUES (`InteractionGroup::NAME = 'name'` and
`ExternalSubtitle::NAME = 'subtitle_name'` used as sibling keys
in the same wildcard set). The fold emits both ClassConstFetch
nodes verbatim — correct, because the AST node identity
disambiguates at the source level (reader sees the host class)
and the resolved values disambiguate at runtime (Laravel sees
distinct keys). Not a collision under case (c) because the
runtime values differ; case (c) is value-collision specifically.

**(c) Const value collisions** — two consts evaluating to the
same string (`InteractionGroup::FOO = 'foo'` AND
`AnotherClass::FOO = 'foo'`, both used as sibling keys in the
same wildcard set). Bail-and-log: the fold can't deterministically
pick which CONST to keep in `children([...])` — both produce key
`'foo'` but reference different declarations. Skip-log message:
`"const value collision in wildcard fold: <CONST_A> and <CONST_B> both evaluate to '<value>'. Refactor to use literal string keys, or remove the duplicate."`.

**(d) Sibling non-wildcard rule shares the inferred parent key**
— if the wildcard set is `'*.' . CONST::A` + `'*.' . CONST::B`
AND a sibling rule has key `'*'` already (or one whose value
equals the inferred wildcard parent), bail. Two competing
top-level `'*'` entries can't coexist; the existing rule wins.
Skip-log: `"wildcard fold target '*' is already used by a non-wildcard sibling rule; folding would clobber it"`.

## 3. Emitter changes

### New output shape: `'*' => array()->children([…])`

The existing `applyGroups()` builds an `each(<array>)` call. The
new `kind: 'string_prefix_const_suffix'` path needs a parallel
`children(<array>)` call. Difference at the AST level:

- `each(...)`: `MethodCall(receiver: array(), name: 'each', args: [Array_])`
- `children(...)`: `MethodCall(receiver: array(), name: 'children', args: [Array_])`

The inner `Array_` items use `ArrayItem(key: ClassConstFetch, value: <rule>)` for
the new form, vs `ArrayItem(key: String_, value: <rule>)` for the
existing.

### Children ordering: source-order, locked

Children in the emitted `children([...])` array **preserve source
order of the input rule keys** — first wildcard sibling key in
source becomes first child, second becomes second, etc.

Rationale: faithful to author intent; deterministic without
needing to explain a sort order in code review; aligns with the
"rector emit is faithful to author intent" framing across the
codebase. Const-name alphabetical or const-value alphabetical
were considered and rejected — the former silently reorders
lines (git blame archaeology cost), the latter sorts by a value
the reader can't see at the call site.

Fixture `wildcard_prefix_concat_source_order_preserved.php.inc`
(in §4 below) uses non-alphabetical const names to pin this
contract. A future refactor introducing sort would fail the
fixture.

### Required: ClassConstFetch preservation

The emitter MUST NOT call `getValue()` or any string-resolution
on the ClassConstFetch. It copies the AST node verbatim into the
generated `ArrayItem`'s key. This:

- Preserves consumer refactor safety (renaming the const value
  reflects in the rector output without re-running the rector).
- Keeps the diff readable on review (the const reference at the
  call site is what the developer reads).
- Avoids the indexer needing `resolveClassConstToString()` —
  string resolution is only used for COLLISION DETECTION (case c
  above), not for the emit.

### Children's import handling

The new `children([…])` emit references `ClassConstFetch` nodes
that already exist in the source (their `use` for the host class
is already present). No new `use` statement needed. The
`ArrayRule::children()` method itself is part of `FluentRule`'s
existing surface; the source already imports `FluentRule` if it
calls `FluentRule::array()`. No import changes required.

## 5. Migration

### For consumers using the new shape pre-0.19

Pre-0.19 the `'*.' . CONST` shape skipped with `concat key too
complex to parse for grouping`. Post-0.19 it folds. Consumers'
rule arrays change from flat-with-concat-keys to the new
`children([...])` form. Pure additive — no consumer code
breaks.

For Validator subclasses (the hihaho case): the 0.17.1 gate
still skips them BEFORE the new fold runs (`qualifiesForShapeChange`
returns false on attribute-only Validator subclasses), so
0.19 doesn't change their behavior. They continue to skip-and-log
with the gate message; the consumer wraps the wildcard rules
manually. The new fold is for FormRequest / fluent-trait /
Livewire consumers using the same shape.

### For consumers writing the existing shape

No change. The existing `<class-const> . '<dotted>'` parser path
is untouched. Existing fixtures pass unchanged.

## Open Questions

1. **Conflict resolution for case (d) — should the rector's bail be configurable (allow-fold-anyway flag)?** No. The bail is correctness-critical; an explicit clobber would silently drop the existing non-wildcard rule. Consumers wanting both should write the rule manually.

## Resolved Questions

1. **Recursive wildcard `'**.' . CONST`?** **Decision:** out of scope for 0.19. **Rationale:** consumer audit (2026-04-27) found zero usage in the dogfood codebase. Reopen on consumer report.
2. **Emit method choice — `children([...])` vs new `eachChildren([...])`?** **Decision:** stay with `ArrayRule::children([...])`. **Rationale:** already public surface on `laravel-fluent-validation`; introducing `eachChildren()` would add new public surface that needs cross-package coordination then deprecation; the `'*' => array()->children([...])` semantic reads cleanly without the new method.
